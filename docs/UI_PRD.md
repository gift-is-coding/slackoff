# Slackoff UI PRD（Command Center 版）v0.1

> 日期：2026-03-14  
> 关联文档：`./PRODUCT_DESIGN.md`

---

## 1. 目标

把前端从“层级管理后台”升级为“多屏监控 + 快捷键驱动”的操作台：

- 全局态势一眼可见（P0/P1、待确认、草稿、执行回执并列）
- 高频动作 2 步内完成
- 大部分核心流程可纯键盘完成

---

## 2. 页面信息架构

### 2.1 顶层页面

1. Command Center（默认首页）
2. Inbox（全量列表）
3. Drafts（草稿池）
4. Queue（待执行）
5. History（执行记录）
6. Policies（策略）
7. Settings（设置）

### 2.2 首页布局（默认）

- 顶部：Global Command Bar（搜索、跳转、批量动作）
- 主区：2x2 Monitor Wall
  - Panel A：P0/P1 实时队列
  - Panel B：待确认执行（Gate B）
  - Panel C：草稿变更流
  - Panel D：执行回执与失败重试
- 右上：状态条（未处理、阻塞、失败）

---

## 3. 核心交互流程

### 3.1 快速处理（主流程）

1) 在左侧列表选中一条（`W/S` 或 `↑/↓`）
2) Enter 打开右侧明细（原文 + AI建议 + 截图）
3) **第一确认：同意方案（Approve Plan）**
   - `Y` 同意方案 / `I` 忽略 / `/` 输入补充指令后 Enter
4) 若动作为外部写操作（发送/发布等），进入执行前校验
5) **第二确认：确认执行（Confirm Execute）**
   - 弹出最终确认面板，展示：目标对象 + 最终内容 + 执行前截图
   - 用户确认后才真正执行

### 3.2 批量处理（运营流）

1) 在 Command Bar 输入筛选（如 `p0 from:boss`）
2) 选择多条 -> `Bulk Approve Plan`
3) 跳转 Queue 面板逐条发送确认

### 3.3 异常恢复

1) Panel D 出现失败项（红）
2) Enter 查看失败原因
3) `R` 重试 / `Esc` 退回草稿 / `Assign` 指派人工

## 4. 快捷键规范（MVP）

- `Cmd/Ctrl + K`：打开全局搜索
- `W/S` 或 `↑/↓`：列表上下移动
- `Enter`：打开详情 / 提交当前输入
- `Esc`：退出当前输入模式 / 关闭弹层
- `Y`：同意方案（第一确认）
- `I`：忽略（低/中风险可直接忽略）
- `/`：进入指令输入模式（输入后 Enter 执行，Esc 取消）
- `E`：编辑 AI 草稿
- `F`：仅看 P0/P1
- `R`：失败项重试
- `?`：快捷键帮助

## 6. 视觉规则

- 仅用颜色表达紧急度：
  - P0 红、P1 橙、P2 蓝、P3 灰
- 卡片默认“最小决策信息”，避免大段正文
- 高风险动作（Send/Delete）必须高对比提示并二次确认

---

## 7. 状态模型（前端）

- Item 状态：`NEW -> TRIAGED -> PLAN_READY -> PLAN_APPROVED -> EXEC_PENDING_CONFIRM -> EXECUTED -> VERIFIED`
- 前端显示规则：
  - `NEW/TRIAGED`：Panel A
  - `PLAN_APPROVED/EXEC_PENDING_CONFIRM`：Panel B
  - `DRAFT_READY`：Panel C
  - `EXECUTED/FAILED`：Panel D

## 8. 可用性验收标准

- 常见任务耗时 ≤ 20 秒
- 80% 高频动作可纯键盘完成

## 10. 人机交互核心模型（Radar + Conveyor）

> 目标：同时满足“看全局”与“快速逐条处理”。

### 10.1 双层注意力架构

- **Layer A：Radar（外围雷达层）**
  - 展示全局态势：总未处理、P0/P1 数、阻塞、失败、SLA 风险。
  - 形式：四象限面板 + 状态条，保持持续可见。
- **Layer B：Conveyor（焦点流水线）**
  - 一次只处理一条“下一最佳项”（Next Best Item）。
  - 形式：焦点卡 + AI建议 + Yes/No 快捷决策。

> 这样用户“眼里有全局，手里只有一条”。

### 10.2 信息密度与重要性平衡

每条消息在列表只保留 1 行最小决策信息：

- 谁（来源/对象）
- 要什么（意图）
- 何时（截止/时效）
- 为什么现在（AI 解释）

其余上下文按需展开（Quick Peek），默认不打扰。

### 10.3 Next Best Item（下一条）计算

建议排序分：

`Score = 影响度 × 紧迫度 × 阻塞度 × 对象权重 + 老化加权`

- 影响度：老板/客户、业务损失、跨团队影响
- 紧迫度：截止时间、是否临近会议
- 阻塞度：是否卡住他人工作
- 对象权重：关键联系人优先
- 老化加权：防止长期积压永远排不到

### 10.4 单条处理卡（Focus Card）

卡片字段（固定顺序）：
1. 标题：来源 + 对象 + 摘要
2. AI建议（只给一个主建议）
3. 解释（1行）
4. 风险标签（Low/Medium/High）
5. 快捷决策按钮：`Yes` / `No`
6. 次级动作：`Edit` / `Snooze` / `Need Info`

### 10.5 Yes / No 交互语义（你提的核心）

- **Yes**：采纳 AI 主建议
  - Low Risk：可直接进入执行
  - Medium/High Risk：进入最终确认 Gate B
- **No**：拒绝 AI 主建议
  - 快速原因标签（可选）：
    - 误判优先级
    - 语气不合适
    - 信息不足
    - 暂不处理
  - 系统记录反馈，用于后续建议优化

### 10.6 节奏设计（快处理，不焦虑）

- 默认进入“处理模式”时，系统持续喂给下一条
- 单条目标处理时长：
  - 简单项：5~8秒
  - 复杂项：20秒内完成决策（处理/延后/转人工）
- 完成后自动跳下一条，减少鼠标和页面跳转

### 10.7 过载保护（Anti-overload）

- 主屏同时可操作事项不超过 7 条（其余折叠）
- 非 P0/P1 默认压缩显示
- 每 10 条后给“批处理暂停点”（可一键回看全局）

### 10.8 关键可观测指标（HCI）

- `Decision Latency`：单条从出现到 Yes/No 的时间
- `AI Accept Rate`：Yes 占比
- `Manual Edit Rate`：采纳前编辑比例
- `Mis-priority Feedback Rate`：误判优先级反馈率
- `Throughput`：每 10 分钟处理条数

这些指标用于持续优化“信息量 vs 可决策性”的平衡。

## 11. 方案 B v2（根据最新反馈升级）

### 11.1 结构调整

- 左侧列表缩窄，右侧明细区放大（从约 1/3 提升到接近 2/5~1/2 视觉权重）
- 左侧仍承担“全局扫视”；右侧承担“单条完整决策”

### 11.2 右侧明细区必备模块（固定顺序）

1. **原文（Source Message）**
   - 展示消息原文（非摘要）
   - 显示来源、对象、线程、截止时间、风险级别
2. **建议回复（AI Draft）**
   - 展示可直接发送草稿
   - 1 行解释“为什么这么建议”
3. **上下文截图（可选）**
   - 可展示当时软件页面截图 / 会话截图
   - 支持点击放大与关闭
4. **决策区（主动作）**
   - `Y 确认`（采纳）
   - `I 忽略`（非关键直接跳过）
   - `/ 指令输入 + Enter`（用户追加指令重写）

### 11.3 Slash 指令交互（轻量命令模式）

- 按 `/` 进入指令输入模式
- 输入补充要求后按 `Enter` 提交，AI 即时重算建议
- 按 `Esc` 退出指令模式（不提交）
- 示例：
  - `/rewrite 更简洁`
  - `/tone 更强势`
  - `/ask 先追问对方可接受截止时间`

### 11.4 忽略（Ignore）动作的边界

- 仅允许在 `LOW/MED` 风险直接忽略
- `HIGH` 风险忽略需二次确认并要求理由（防漏关键）

### 11.5 你可能还没提到，但建议补上的点

1. **Undo（可撤销）**
   - 发送后 5~15 秒可撤销（若渠道能力支持）
2. **信心分数 + 不确定提示**
   - 当 AI 不确定时，主动显示“信息不足”并请求补充
3. **最小证据链**
   - 每次 Yes/Ignore/改写都保留决策记录，便于复盘
4. **下一条预取（Prefetch）**
   - 当前项决策时后台预加载下一条，减少等待
5. **防误触保护**
   - Yes/Ignore 支持可配置“长按 300ms”或二次按键确认

### 11.6 双确认机制（核心安全流）

- **确认 1：同意方案（Approve Plan）**
  - 目的：确认“AI理解和处理方向”
  - 输入：`Y / I / /指令`
  - 输出：形成“待执行草案”
- **确认 2：确认执行（Confirm Execute）**
  - 触发条件：外部写操作（发送/发布/删除等）
  - 必看信息：
    1) 目标对象（发给谁）
    2) 最终内容（发送什么）
    3) 执行前截图（在什么上下文发送）
  - 只有第二次确认通过，才执行

## 12. 方案 B v3（按最新要求修订）

### 12.1 明确去掉的内容

- 去掉“右下角风格化说明区”
- 不再强调某个产品风格标签

### 12.2 保留并强化的交互

- Slash 输入模式：
  - `/` 进入输入
  - `Enter` 提交
  - `Esc` 退出
- 导航改为：`W/S` 或 `↑/↓`
- 两次确认：
  1) 同意方案（Approve Plan）
  2) 确认执行（Confirm Execute）

### 12.3 执行前截图确认（单截图版）

执行前确认不再拆成三个信息框，改为 **一张完整执行前截图**：

- 截图中必须可见：收件人、最终文案、发送按钮/关键动作位
- 系统仅在截图上做轻量高亮（框选），不额外拆分三栏
- 用户在这一个截图视图中完成最终核对

用户确认后才触发真正发送。

### 12.4 比例与布局建议（v3.1）

- 左右比例建议：`40 : 60`（左侧全局、右侧决策）
- 右侧内部建议：
  - 顶部阶段条 + 原文 + AI建议：约 45%
  - 底部执行确认（单截图）：约 55%
- 原则：让“执行前截图”拥有足够面积，确保肉眼可直接核对关键信息

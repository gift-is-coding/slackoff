# Slackoff（摸鱼）产品设计方案 v0.1

> 日期：2026-03-14  
> 定位：基于 OpenClaw 的「工作消息分诊 + 智能代办 + 人工确认执行」操作台

---

## 1. 产品愿景（Vision）

让白领在不漏关键事项的前提下，**把注意力从“被消息推着走”切换到“按优先级主动处理”**。  
Slackoff 不是“自动帮你乱回消息”，而是一个有边界的执行副驾：

- 帮你看全局：汇总所有渠道、自动高亮重点
- 帮你想方案：给出回复草稿和行动建议
- 帮你落执行：在你确认后高质量执行
- 坚持可控：所有有外部影响的动作都要最终确认

---

## 2. 核心问题（Problem Statement）

典型白领每天面临：

1. **多渠道轰炸**：IM、邮件、群聊、通知中心分散
2. **优先级混乱**：紧急和不紧急混在一起
3. **回复成本高**：每条消息都要“重新理解+组织语言”
4. **执行碎片化**：回复、跟进、建任务、查资料在不同工具间来回跳
5. **心理负担大**：怕漏老板/客户重要消息，无法安心“摸鱼/深度工作”

---

## 3. 产品目标与非目标

### 3.1 目标（Goals）

- G1：统一收敛消息输入，形成单一处理入口（Unified Inbox）
- G2：自动识别“必须处理”的消息并解释原因
- G3：自动生成高质量回复草稿与可执行计划
- G4：所有外部影响动作（发消息/发邮件/删改内容）必须人工最终确认
- G5：通过前端操作台形成可视化、可追踪、可回溯的工作流

### 3.2 非目标（Non-goals，MVP阶段）

- 完全无人值守的自动回复机器人
- 企业级组织权限系统（先做单用户/小团队）
- 全渠道深度双向同步（先优先高频渠道）

---

## 4. 目标用户（Target Users）

1. **高消息负载白领**（产品、运营、销售、客户成功、管理者）
2. **远程协作人员**（跨时区、多群组沟通）
3. **需要“低心智负担”工作流的人**（希望先看重点，再决定是否处理）

---

## 5. 产品原则（Design Principles）

1. **Human-in-the-Loop（人类在环）**：
   - 读和分析可自动
   - 草拟可自动
   - 有外部影响动作必须二次确认
2. **Explainability（可解释）**：每条高亮都要说明“为什么重要”
3. **Progressive Automation（渐进自动化）**：先建议，再半自动，再受控执行
4. **Safety by Default（默认安全）**：默认不发送、不发布、不删除
5. **One Console（单一操作台）**：尽量在一个界面完成分诊→决策→执行

---

## 6. 功能架构（What Slackoff Does）

## 6.1 统一消息收件箱（Unified Inbox）

- 聚合来源：通知中心、IM、邮件、协作工具
- 标准化展示：来源、联系人、主题、摘要、时间、风险级别
- 去重与聚类：同一会话/同一主题自动合并

## 6.2 智能优先级引擎（Priority Engine）

为每条消息打分并分层：

- **P0 立刻处理**（高影响/强时效）
- **P1 今日处理**
- **P2 本周处理**
- **P3 可忽略/仅归档**

评分信号（可配置）：
- 发件人重要度（老板/关键客户/核心同事）
- 截止时间或会议临近
- 被@、被点名、重复催促
- 消息情绪和紧迫语义（“ASAP”“blocked”等）
- 历史上下文（相同主题是否未完成）

## 6.3 行动建议与草稿（AI Action Planner + Drafting）

对每条待处理项自动给出：

- 建议动作：回复 / 延后 / 建任务 / 约会 / 升级给他人
- 建议优先级与截止时间
- 回复草稿（可选风格：简短、专业、友好、强势）
- 所需补充信息清单（如需要更多上下文）

## 6.4 双重确认执行（Approval Gates）

### Gate A：计划确认
用户确认“做什么”（批量）

### Gate B：执行确认（关键）
在外部影响动作前二次确认“怎么做”
- 例：发送消息、发送邮件、创建会议邀请、删除内容

> 关键规则：**未确认 = 不执行**

## 6.5 执行与回执（Execution + Verification）

- 执行后自动回写状态：已发送、已失败、待重试
- 保存证据：执行时间、目标对象、最终内容、执行结果
- 可追溯日志：满足“出了问题可复盘”

## 7. 前端操作台设计（Console UX）

## 7.1 信息架构（IA）

左侧导航：
1. Inbox（全部消息）
2. Needs Attention（高优先）
3. Drafts（草稿）
4. Tasks（待办）
5. Queue（待执行）
6. History（执行记录）
7. Policies（自动化策略）
8. Settings（渠道与偏好）

## 7.2 三栏布局（推荐）

- **左栏**：消息列表（带优先级彩色标签）
- **中栏**：当前消息详情 + 上下文线程
- **右栏**：AI建议面板（建议动作、草稿、风险、确认按钮）

## 7.3 卡片字段（每条消息）

- 来源（Teams/Slack/Email/WeChat等）
- 联系人/群组
- 摘要 + 原文快捷查看
- 优先级（P0~P3）
- 建议动作（Reply/Task/Snooze）
- 推荐草稿（可编辑）
- 风险等级（Low/Medium/High）
- 解释（为何被高亮）

## 7.4 核心交互

- 一键“仅看P0/P1”
- 批量确认计划（Approve Plan）
- 单条编辑草稿（Inline Edit）
- 发送前预览Diff（AI稿 vs 人工改稿）
- 关键动作弹窗二次确认（含目标对象、内容、渠道）

## 7.5 前端改版方向（Command Center / 多屏监控风格）

> 设计目标：从“层级导航式管理后台”转向“全局可见 + 快捷键驱动 + 一跳执行”的操作台。

### 布局策略

- 默认首页采用 **2x2 / 3x2 多面板（Monitor Wall）**，每个面板承载一个关键流：
  - P0/P1 实时队列
  - 待确认执行（Gate B）
  - 草稿变更流（最近编辑）
  - 执行回执/失败重试
- 顶部保留 **全局 Command Bar**（支持搜索、跳转、批量动作）
- 右上角保留 **全局状态条**（未处理数量、阻塞数、失败数）

### 交互策略（去层级化）

- 弱化“回收站 / 消息站”式深层菜单，改为：
  - 面板内就地处理（Inline）
  - 侧滑详情（Quick Peek）
  - 键盘弹层（Command Palette）
- 所有高频动作保证 **2 步内完成**（选中 -> 执行/确认）

### 快捷键系统（首版建议）

- `1~6`：切换主面板焦点
- `J/K`：面板内上下移动
- `Enter`：打开详情
- `A`：Approve Plan
- `S`：Send（触发最终确认）
- `E`：编辑草稿
- `F`：仅看 P0/P1 过滤
- `Cmd/Ctrl + K`：打开命令面板

### 可视化优先级

- 颜色只表达紧急程度（P0 红 / P1 橙 / P2 蓝 / P3 灰）
- 面板卡片默认只显示“决策所需最小信息”：来源、对象、摘要、建议动作、风险
- 复杂上下文通过悬浮或侧滑按需展开，避免主界面信息噪音

### 体验验收标准

- 新用户 5 分钟内可完成首个“从分诊到发送”的闭环
- 常见任务（查看 -> 轻改草稿 -> 发送）≤ 20 秒
- 不依赖鼠标也能完成 80% 高频操作

## 8. 系统设计（OpenClaw-Based Architecture）

## 8.1 逻辑分层

1. **Ingestion Layer（采集层）**
   - 读取通知与消息源
2. **Understanding Layer（理解层）**
   - 分类、聚类、优先级评分、意图识别
3. **Planning Layer（规划层）**
   - 生成动作建议与草稿
4. **Approval Layer（审批层）**
   - Gate A / Gate B 状态机
5. **Execution Layer（执行层）**
   - 调用 OpenClaw 工具执行外部动作
6. **Audit Layer（审计层）**
   - 全链路日志与可追溯记录

## 8.2 与 OpenClaw 能力映射

- 消息发送：`message` 工具
- 通知读取：`nodes.notifications_list` / 已有通知读取链路
- 调度与提醒：`cron`
- 跨会话编排：`sessions_spawn` / `sessions_send`
- 浏览器执行：`browser`
- 文件与知识落地：`read/write/edit`

## 8.3 Agent 角色拆分（建议）

- **Triage Agent**：分诊与优先级
- **Draft Agent**：草稿生成
- **Action Planner Agent**：动作规划
- **Safety Agent**：策略校验（是否需最终确认）
- **Executor Agent**：执行工具调用
- **Verifier Agent**：结果验证与失败恢复

---

## 9. 工作流状态机（State Machine）

`NEW -> TRIAGED -> PLAN_READY -> PLAN_APPROVED -> DRAFT_READY -> EXEC_PENDING_CONFIRM -> EXECUTED -> VERIFIED -> CLOSED`

异常分支：
- `REJECTED`（用户拒绝）
- `NEEDS_INFO`（上下文不足）
- `FAILED`（执行失败，待重试）

每次状态变更都记录：
- 操作人（用户/系统）
- 时间戳
- 变更前后差异
- 关联证据

---

## 10. 安全与权限模型（Critical）

## 10.1 动作分级

- **L0 只读**：读取消息、分类、总结（可自动）
- **L1 低影响写**：生成草稿、建本地待办（可自动）
- **L2 外部影响写**：发送消息/邮件、创建会议（必须最终确认）
- **L3 高风险改动**：删除/批量改写/公开发布（双重确认 + 明示风险）

## 10.2 执行策略

- 默认 deny-write（写操作默认禁用，需显式批准）
- 单次授权优先（不要长期“免确认”）
- 敏感对象白名单（重要联系人默认更严格）
- 全量审计日志（合规与复盘）

---

## 11. 数据模型（MVP）

核心表（或集合）：

1. `source_accounts`：接入渠道账户
2. `message_items`：标准化消息实体
3. `threads`：会话与主题聚类
4. `priority_scores`：评分与解释
5. `action_plans`：建议动作与状态
6. `drafts`：草稿版本（含编辑历史）
7. `approvals`：审批记录（Gate A/B）
8. `executions`：执行请求与回执
9. `policies`：自动化与安全策略
10. `user_preferences`：语气、工作时间、重点联系人

## 12. MVP范围建议（8~12周）

## Phase 1（2周）：基础可用
- 单渠道接入（优先邮件或IM之一）
- Inbox + P0/P1 高亮
- 手动触发草稿

## Phase 2（2~3周）：核心闭环
- 建议动作 + 自动草稿
- Gate A/Gate B 审批流
- 发送执行 + 回执

## Phase 3（2~3周）：多渠道与策略
- 第二、第三渠道接入
- 优先级规则可配置
- 批量处理与模板回复

## Phase 4（2~4周）：产品化打磨
- 日志审计页
- 失败重试与可恢复机制
- 指标看板与体验优化

---

## 13. 指标体系（Success Metrics）

北极星指标（建议）：
- **每用户每日节省的消息处理时间（分钟）**

关键指标：
- 重要消息漏处理率（越低越好）
- AI草稿采纳率（直接发送/小改后发送）
- 平均从收到到处理完成时长
- 用户确认后的执行成功率
- 用户对“高亮准确性”的满意度

---

## 14. 风险与对策

1. **误判优先级**
   - 对策：可解释评分 + 一键纠偏 + 用户规则学习
2. **草稿语气不合适**
   - 对策：个性化语气模板 + 历史偏好学习
3. **误发风险**
   - 对策：强制最终确认 + 目标对象高亮 + 冷静期（可选）
4. **多渠道稳定性**
   - 对策：先高频渠道，逐步接入；失败可回退人工
5. **隐私与合规**
   - 对策：最小权限、数据加密、可配置保留期限

## 15. 你这个想法最关键的产品差异点（Differentiators）

和普通“AI回复助手”相比，Slackoff 的差异应强调：

1. **不是只会写字，而是“分诊 + 规划 + 审批 + 执行”全链路**
2. **不是黑盒自动化，而是可解释、可控、可追责**
3. **不是单聊天工具插件，而是跨渠道统一操作台**
4. **不是替你做决定，而是让你快速做对决定**
